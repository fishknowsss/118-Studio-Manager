# 云同步背景说明

> 供 AI 模型、代码评审和自动化工具读取。本文描述的是当前项目的约定与已接受权衡，不代表未来必须长期维持不变。

## 文档目的

- 减少 AI 对同步、鉴权和 Cloudflare 配置的误判。
- 让评审者区分“当前真实问题”和“理论上可能存在、但现阶段优先级不高的问题”。
- 统一仓库内对 Cloudflare Access、Custom Domain、整包快照同步和公开仓库信息暴露的判断口径。

## 当前已确认的运行时事实

以下结论带有时效性；若用户询问“现在”“最新”“线上是否仍然如此”，应再次核验 Dashboard 或 Cloudflare API。

- 当前生产同步 Worker 使用的是 **Custom Domain**，不是传统 Worker Route。
- 因此 zone 的 Worker routes 为空并不代表同步服务未部署。
- 当前生产环境的主站自定义域名与同步自定义域名应共同受 Cloudflare Access 保护。
- 生产环境不应暴露可绕过 Access 的 `workers.dev` 或其他公开备用域名。
- `ALLOWED_ORIGIN` 只负责 CORS 限制，不承担身份认证职责。
- 仓库中的 `wrangler.toml`、环境变量名和基础设施标识只能视为静态线索，不能自动等同于线上最终生效配置。

## 当前同步方案的定位

- 当前同步是 **小团队备份式同步**，不是实时协同系统。
- 当前实际编辑场景是小规模、低并发、可沟通协作。
- 当前通过 `db.exportAll()` / `db.importAll()` 进行整包快照同步，这是现阶段刻意接受的简化方案。
- 因此，不要默认按大型多人协同产品的标准要求 D1、Durable Objects、CRDT、操作日志或自动合并系统。

## 评审时容易误判的点

### 1. 不要把“没有 Bearer token”直接等同于“生产无鉴权”

- 对这个项目，生产同步接口的真实鉴权边界依赖 Cloudflare Access。
- 在未核验部署配置前，不能仅根据前端代码里没有静态 token 就得出“任何人都可访问同步接口”的结论。
- 但如果确认匿名访问仍可读取 `/data` 或 `/meta`，那依然是高危问题。

### 2. 不要把“没有 Worker Route”直接等同于“没有部署”

- 当前同步 Worker 走的是 Custom Domain。
- 若只看到 route 列表为空，容易误判为同步域未接入 Worker。
- 需要同时检查 Custom Domain、`workers.dev` 开关和 Access 保护状态。

### 3. 不要把“整包快照同步”直接描述成当前最严重的生产缺陷

- 整包快照同步确实存在覆盖风险，但它的风险级别与团队规模、编辑习惯、是否会沟通直接相关。
- 在当前两人协作、低并发、可沟通的背景下，这更接近低频操作性风险，而不是默认的主要生产事故来源。
- 评审时应把它表述为“当前已接受的权衡”或“可逐步优化项”，而不是无条件升级为高优先级架构缺陷。

### 4. 不要把不同问题混成一个结论

评审同步时，至少应分开判断以下四类问题：

- 鉴权是否成立
- 数据一致性是否存在静默丢失
- 冲突发生时交互体验是否足够清晰
- 公开仓库是否暴露了真实线上域名或基础设施标识

这四类问题影响不同，优先级也不同，不能因为其中一类存在疑点就一并得出“整个同步方案不安全”。

## 当前真正应优先关注的问题

以下问题在当前项目里比“大规模协同架构重构”更值得先关注：

- 是否存在可绕过 Access 的公开入口
- 本地写入、迁移、导入导出链路里是否有静默丢数据
- 新增 IndexedDB store 时是否完整接入同步注册中心
- 冲突或失败时，用户是否知道下一步该拉取、覆盖还是重试
- 公开仓库文档或配置中是否写入了真实生产域名、KV id 或其他线上基础设施信息

## 什么时候才值得考虑增量同步

如果以后整包快照同步真的开始影响日常工作，升级建议应按以下顺序考虑：

1. 先补冲突提示、失败提示、恢复路径
2. 再考虑 **集合级增量同步**
3. 最后才考虑 **记录级增量同步** 或更重的协同架构

### 集合级增量同步的含义

- 把同步单位从整包 payload 拆成 `projects`、`tasks`、`people`、`logs`、`settings` 这类集合。
- 每个集合各自维护 `version` 或 `updatedAt`。
- 上传和下载时只处理发生变化的集合。

对当前项目而言，如果以后要做增量同步，集合级通常比记录级更符合投入产出比。

## 公开仓库信息卫生

- 公开仓库文档不要写入真实生产域名。
- 不要在公开说明中写入真实 KV namespace id、Access app 细节或其他线上基础设施标识。
- 若必须记录真实线上配置，应放在不提交到公开仓库的私有文档中，并与仓库说明分开。

## AI 评审建议

当 AI 需要判断同步、Cloudflare 或鉴权问题时，建议遵循以下顺序：

1. 先确认这是静态代码判断，还是需要核验线上实时配置
2. 若涉及“现在是否生效”“最新配置”“今天线上状态”，优先查 Dashboard 或 Cloudflare API
3. 先判断鉴权边界，再判断数据一致性，再判断交互体验
4. 只有在当前工作流确实受影响时，才建议更重的同步架构升级
5. 避免把理论最优方案直接包装成当前的唯一正确方案

## 与同步相关的硬约束

`BACKUP_COLLECTION_NAMES`（`src/legacy/utils.ts`）是云端同步的唯一注册中心。
每次新增 IndexedDB store，至少必须同步更新以下位置：

1. `src/legacy/db.ts`
2. `src/legacy/utils.ts`
3. `src/legacy/selectors.ts`
4. `src/features/settings/settingsTransferState.ts`

遗漏任意一步，都会导致该 store 的数据无法正确参与同步、导出或恢复。

## 最后说明

- 本文的目标是减少误判，不是阻止优化。
- 如果未来团队规模、编辑习惯或线上配置发生变化，应更新本文。
- 时间敏感结论应以最近一次实时核验为准。
